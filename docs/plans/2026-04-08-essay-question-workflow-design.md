# Essay Question Workflow Redesign

## Goal

ForJob의 자기소개서 작성 흐름을 `프로젝트 기반 문항 작성 워크플로우`로 재설계한다.

핵심 목표는 다음과 같다.

1. 프로젝트 탭에서 이미 수집한 자소서 문항을 자기소개서 탭이 그대로 재사용한다.
2. 자기소개서 탭은 더 이상 문항을 다시 입력받지 않는다.
3. 사용자는 문항 1부터 순서대로 작성하되, 언제든 문항 1/2/3 사이를 전환할 수 있다.
4. 각 문항은 `완료` 버튼으로 고정할 수 있고, 완료된 답안은 프로젝트 컨텍스트에 포함된다.
5. 완료된 문항도 다시 열어서 수정할 수 있다.

## Current State

현재 흐름은 두 군데에서 문항을 다룬다.

- 프로젝트 탭은 `essayQuestions: string[]`를 받지만, UI는 하나의 `textarea`에 여러 줄을 넣는 방식이다.
- 자기소개서 탭은 `run-question` 필드로 질문을 다시 입력받는다.

이 구조의 문제:

- 같은 문항을 두 번 입력해야 한다.
- 프로젝트에서 이미 확보한 문항/인사이트와 자기소개서 작성 흐름이 분리돼 있다.
- 문항별 작성 상태를 추적하기 어렵다.
- `/done` 같은 대화형 종료 방식은 문항 단위 작업 완결감이 약하다.

## Design Principles

- 질문은 프로젝트가 소유한다.
- 답안은 문항별로 분리 저장한다.
- 문항별 작성 흐름은 순차 진행을 기본으로 하되, 사용자가 자유롭게 왕복할 수 있어야 한다.
- `완료`는 세션 종료가 아니라 `문항 답안 고정`을 의미한다.
- 고정된 답안도 나중에 수정 가능해야 한다.
- 현재 저장소/컨텍스트 구조를 최대한 재사용한다.

## Alternatives Considered

### Option A: Minimal patch

- 프로젝트 탭 문항 입력만 배열형으로 바꾸고
- 자기소개서 탭은 기존 `run-question`을 자동 채우기만 한다.

장점:

- 가장 빠르다.

단점:

- 문항별 상태와 완료 흐름이 여전히 모호하다.
- 자기소개서 탭이 여전히 “실행 폼”에 가깝다.

### Option B: Project-owned question workflow

- 프로젝트 탭은 문항 배열 UI로 바꾸고
- 자기소개서 탭은 선택된 프로젝트의 문항 목록을 기반으로 동작한다.
- 각 문항은 자체 답안 draft와 완료 상태를 가진다.
- `완료` 버튼으로 컨텍스트에 포함시키고 자동으로 다음 문항으로 이동한다.

장점:

- 지금 제품 방향과 가장 잘 맞는다.
- 인사이트 → 문항별 작성 → 프로젝트 컨텍스트 누적 흐름이 자연스럽다.
- 현재 `essayQuestions[]`, project document storage, context compiler를 그대로 활용할 수 있다.

단점:

- 자기소개서 탭의 UI/state를 꽤 손봐야 한다.

### Option C: Full writing board

- 자기소개서 탭을 완전히 문항별 작업보드로 재설계한다.
- 각 문항이 자체 run history, status, draft, final answer, completion metadata를 모두 가진다.

장점:

- 장기적으로 가장 완성도 높다.

단점:

- 이번 단계 범위로는 크다.

## Recommendation

`Option B`를 채택한다.

이유:

- 현재 ForJob는 이미 프로젝트가 문항과 인사이트를 보유한다.
- 이제 자기소개서 탭도 그 구조를 따라가야 한다.
- 하지만 전체 글쓰기 보드 제품으로 한 번에 점프할 필요는 없다.
- 문항별 답안 문서와 완료 상태만 도입해도 UX 개선 효과가 크다.

## UX Redesign

## 1. Projects Tab

`에세이 문항` 입력을 단일 textarea에서 `배열형 입력`으로 바꾼다.

새 동작:

- 기본적으로 빈 입력칸 1개를 보여준다.
- 각 칸은 문항 하나만 입력한다.
- `문항 추가` 버튼을 누르면 새 빈 칸이 아래에 추가된다.
- 각 문항 칸은 삭제 가능하다.
- 저장 시 빈 값은 제거하고 `essayQuestions: string[]`로 직렬화한다.

이 변경의 목적:

- 문항을 입력 시점부터 독립된 단위로 다루게 만든다.
- 이후 자기소개서 탭에서 같은 순서와 인덱스를 그대로 사용할 수 있게 한다.

## 2. Essay Tab Entry

자기소개서 탭은 더 이상 `자소서 질문` 입력 칸을 렌더하지 않는다.

대신:

- 선택된 프로젝트가 없으면 프로젝트 선택 유도
- 문항이 없으면 프로젝트 탭에서 문항 추가 유도
- 문항이 있으면 자동으로 `문항 1`을 활성화

상단에는 문항 전환 UI를 둔다.

예:

- `문항 1`
- `문항 2`
- `문항 3`

각 탭에는 상태 chip이 붙는다.

- `미작성`
- `작성 중`
- `완료`

기본 이동 규칙:

- 처음 진입 시 `문항 1`
- `완료` 클릭 시 자동으로 다음 미완료 문항으로 이동
- 사용자는 언제든 이전 문항으로 돌아갈 수 있음

## 3. Question Workspace

문항별 작업 영역은 다음 요소를 가진다.

- 현재 문항 텍스트
- 관련 인사이트 요약 진입점
  - 기업 분석
  - 직무 분석
  - 지원 전략
  - 문항 분석
- 현재 답안 편집 영역
- 실행 버튼
- 완료 버튼

중요한 변화:

- 질문은 프로젝트에서 읽고, 실행 시 자동 주입한다.
- 현재 답안은 문항별 저장 draft에서 불러온다.
- 실행은 선택된 문항 하나를 대상으로만 수행한다.

## 4. Completion Flow

`완료` 버튼은 `/done`을 대체한다.

의미:

- 현재 문항 답안을 “프로젝트 문항 답안”으로 고정한다.
- 이 답안은 이후 프로젝트 컨텍스트에 포함된다.

완료 클릭 시 동작:

1. 현재 문항 답안 저장
2. 문항 상태를 `완료`로 변경
3. 프로젝트 컨텍스트에 포함되도록 처리
4. 다음 문항으로 자동 이동

만약 마지막 문항이면:

- 자동 이동 대신 완료 상태 유지
- 사용자가 다른 문항을 선택할 수 있게 둔다

## 5. Editing After Completion

완료된 문항도 다시 열어 수정할 수 있다.

권장 규칙:

- 완료 문항을 편집하면 즉시 완료 해제하지는 않는다.
- 대신 `수정됨` 상태를 별도로 두지 않고, 저장되지 않은 편집이 있으면 UI에서만 dirty 상태를 보여준다.
- 사용자가 다시 `완료 업데이트`를 누르면 완료본이 갱신된다.

이유:

- 완료 상태가 너무 쉽게 흔들리면 사용자가 진행 상황을 잃는다.
- 프로젝트 컨텍스트에 포함되는 것은 마지막으로 완료된 버전이어야 한다.

## Data Model

현재 `essayQuestions: string[]`는 유지한다.

추가가 필요한 것은 `문항별 답안 상태`다.

권장 구조:

- 프로젝트 레벨 상태에 질문 배열은 그대로 유지
- 답안은 문항별 project-scoped document로 저장
- 문항 상태 메타는 프로젝트 레코드에 별도 저장

예시 메타:

- `essayAnswerStates?: Array<{ questionIndex: number; status: "idle" | "drafting" | "completed"; completedAt?: string; documentId?: string; lastRunId?: string }>`

또는 동등한 map 구조

## Answer Storage Model

각 문항의 답안은 프로젝트 문서로 저장한다.

예시 문서 제목:

- `essay-answer-q1.md`
- `essay-answer-q2.md`
- `essay-answer-q3.md`

권장 이유:

- 이미 project document manifest / preview / pinning 구조가 있다.
- 컨텍스트 컴파일러가 기존 방식으로 읽을 수 있다.
- 사용자가 나중에 문서로도 열어볼 수 있다.

문서 note에는 문항 정보와 생성 출처를 기록한다.

예:

- `Answer for essay question 1. Completed from essay workspace.`

## Context Inclusion

완료된 문항 답안만 프로젝트 컨텍스트에 포함한다.

규칙:

- `미작성` / `작성 중` 답안은 자동 포함하지 않는다.
- `완료` 상태 문항 문서는 기본 포함 또는 강제 포함 방식으로 컨텍스트에 들어간다.
- 현재 작성 중인 문항의 draft는 실행 시 명시적으로 request draft로만 사용한다.

이유:

- 미완성 초안이 다른 문항 작성 컨텍스트를 오염시키지 않도록 한다.
- 완료본만 누적되어 프로젝트의 “검증된 자산”이 되게 한다.

## Run Behavior

자기소개서 탭의 실행은 여전히 기존 review orchestrator를 사용한다.

하지만 입력값이 달라진다.

### Current

- question: 사용자가 직접 입력
- draft: 사용자가 직접 입력

### New

- question: 선택된 프로젝트 문항에서 자동 주입
- draft: 해당 문항의 현재 편집 draft에서 주입

즉 `run-question` 필드는 제거하고, `RunRequest.question`은 선택 문항에서 채운다.

## Conversation Flow

대화형 실행은 유지할 수 있다.

다만 문항을 완료하는 UX는 대화 입력창의 `/done`이 아니라 상위 UI의 `완료` 버튼이 담당한다.

권장 규칙:

- 대화창은 계속 피드백/수정 요청 용도
- 문항 완료는 별도 버튼
- `/done`은 더 이상 기본 안내 문구로 노출하지 않음

필요하면 내부 호환성 차원에서 남겨둘 수 있지만, 사용자 주 흐름에서는 제거한다.

## UI State

웹뷰에 아래 상태가 필요하다.

- `activeQuestionIndex`
- `questionDrafts` 또는 현재 편집 draft 복원 정보
- `questionStatusByIndex`
- `dirtyQuestionIndices`

질문이 바뀌어도:

- 현재 draft가 유지되어야 하고
- 다시 돌아오면 같은 편집 상태가 복원되어야 한다.

## Error Handling

- 프로젝트에 문항이 없으면 자기소개서 탭에서 실행 불가
- 선택 프로젝트가 없으면 자기소개서 탭에서 안내만 표시
- 답안 문서 저장 실패 시 완료 처리하지 않음
- 다음 문항 자동 이동 실패 시에도 현재 문항 완료 상태는 유지

## Testing Strategy

필수 테스트:

1. 프로젝트 탭 문항 배열 입력 직렬화/역직렬화
2. 자기소개서 탭이 프로젝트 문항 1을 기본 선택하는지
3. 문항 탭 전환 시 question source가 프로젝트 문항으로 바뀌는지
4. `run-question` 입력 제거 후 실행이 정상 동작하는지
5. 완료 버튼이 문항 답안 문서를 저장하는지
6. 완료 버튼이 상태를 `완료`로 바꾸고 다음 문항으로 이동하는지
7. 완료된 문항 문서만 프로젝트 컨텍스트에 포함되는지
8. 완료된 문항을 다시 수정하고 `완료 업데이트`할 수 있는지

## Non-goals

- 문항별 개별 run history를 완전한 작업보드 수준으로 노출하는 것
- 문항별 승인 워크플로우, 버전 diff UI
- 전체 자기소개서를 한 문서로 자동 합치는 출판 흐름
- 문항별 별도 메인 에디터 패널 분리

## Success Criteria

- 프로젝트 탭에서 문항을 한 칸씩 추가하는 방식으로 입력할 수 있다.
- 자기소개서 탭에서 문항을 따로 다시 입력하지 않는다.
- 자기소개서 탭은 문항 1부터 시작하고, 문항 1/2/3을 자유롭게 오갈 수 있다.
- 각 문항은 `완료` 버튼으로 고정할 수 있다.
- 완료된 문항 답안은 프로젝트 컨텍스트에 포함된다.
- 완료된 문항도 다시 수정하고 갱신할 수 있다.
- 사용자는 `/done` 없이도 문항 단위로 작업을 끝냈다는 감각을 얻는다.
